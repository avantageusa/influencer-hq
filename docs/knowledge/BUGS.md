# Bug Audit

### [CRITICAL] Plaintext password written to server error log on every login and registration
**File:** `inc/influencer-auth-handler.php:32,35,192,196`

`error_log('temp influencer_register_ajax called: ' . print_r($_POST, true))` executes unconditionally on every login and registration AJAX call. `$_POST` at this point contains the user's raw password. This is not conditional on a debug flag — it fires in production on every request.

**Why confident:** The calls are at the top of both `ihq_login_ajax()` and `ihq_register_ajax()`, before any processing. `print_r($_POST, true)` returns all submitted fields including `password`. The word "temp" in the log prefix confirms it was supposed to be removed.

**Fix:** Delete all four `error_log()` calls prefixed with `"temp"` in that file.

### [CRITICAL] Hardcoded test authorization token sent to production OAuth endpoint
**File:** `inc/influencer-auth-handler.php:282`

The OAuth start-session POST sends `Authorization: milos_testing` as a literal string header to the production AWS API endpoint. This is a developer test credential embedded in source, not a configurable secret.

**Why confident:** The string `milos_testing` is a named test token (developer first name), not a variable reference. The endpoint it reaches is the same production URL used everywhere else in the theme.

**Fix:** Move to a `wp-config.php` constant (`IHQ_OAUTH_SECRET`) and reference it via `defined('IHQ_OAUTH_SECRET') ? IHQ_OAUTH_SECRET : ''`. Rotate the token after the fix.

### [HIGH] JWT sub claim extracted from token with no signature verification
**File:** `inc/api-ajax-calls.php:102–108`

`_challenge_get_sub()` base64-decodes the JWT payload and returns the `sub` field with zero signature validation. The token is read from `ihq_id_token` user meta. Any code path that can write user meta (WP admin, other plugins, a compromised admin account) can inject a crafted token that redirects AWS API calls to a different account's `sub`.

**Why confident:** The function contains no call to a JWT library, no HMAC check, no public-key verification. The comment block is absent. The decoded `sub` value is then used to construct API requests downstream.

**Fix:** Verify the JWT signature against the IdP's public key before trusting the payload, or extract `sub` from the API's own response rather than re-parsing the stored token client-side.

### [HIGH] `_debug` object containing JWT sub and internal user ID returned to every API caller
**File:** `inc/api-ajax-calls.php:114–124`

`_challenge_send_response()` always appends `$debug` to the JSON success response. `$debug` includes: the full AWS request URL, the JWT `sub` claim, and the internal user identifier `influencerhq-wpu-{wp_user_id}`. This is sent to every authenticated browser making a challenge API call, regardless of environment.

**Why confident:** The append is unconditional (`$data['_debug'] = $debug`), not wrapped in any `WP_DEBUG` or environment check. The client-side code in `page-portal-challenges.php:681–747` stores and surfaces this data.

**Fix:** Gate the `_debug` append behind `defined('WP_DEBUG') && WP_DEBUG`, or remove it entirely.

### [MEDIUM] Email delivery silent failure when `wp_mail_failed` hook fires before it is registered
**File:** `inc/email-verification-handler.php:131–142`

The code registers a `wp_mail_failed` callback, calls `wp_mail()`, then removes the callback. If `wp_mail()` fires `wp_mail_failed` synchronously before the variable `$mail_error` is set in the closure (e.g., due to hook priority ordering), `$mail_error` remains `null` and the user receives the message `"Failed to send email"` with no diagnostic detail — and the error log entry is equally uninformative.

**Why confident:** The closure assigns to `$mail_error` via `use (&$mail_error)`. If `wp_mail` internals fire the failed hook before the outer variable reference resolves (race in hook priority), the reference captures `null`. This is a known pattern issue with closures and WordPress hook timing.

**Fix:** Initialize `$mail_error = 'Unknown mail error'` before registering the hook so the fallback is always meaningful. Log the full `WP_Error` object, not just the message string.
